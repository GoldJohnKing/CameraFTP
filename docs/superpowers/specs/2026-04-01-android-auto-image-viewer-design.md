# Android 前台自动显示新图片设计文档

**日期**: 2026-04-01
**版本**: 1.0

---

## 1. 目标

为 Android 端新增一个仅对内置图片查看器生效的能力：

- 当 App 进程处于可见状态时，每次接收到新图片后，自动显示最新收到的图片
- 若内置图片查看器已经打开，则直接跳转到最新收到的图片，而不是再打开一个新的查看器层级
- 在配置界面增加该能力的开关
- 当图片打开方式配置为外部查看器时，隐藏该开关，并且该能力始终不启用

### 1.1 用户确认的行为约束

- 连续收到多张图片时，**总是跳到最新收到的那张**
- “App 处于前台”按 **进程可见** 判定，包括：
  - 正常前台
  - 分屏可见
  - 浮窗 / 多窗口可见
- 仅在“收到图片时 App 可见”这一时刻触发自动显示，不补队列、不延迟补开

---

## 2. 现状

当前代码中已经具备以下基础能力：

- Android 图片打开方式配置已经存在：`built-in-viewer` / `external-app`
- 配置界面已经有 Android 图片查看设置入口
- Android 在图片入库完成后会发出 `gallery-items-added` 浏览器事件
- 前端图库已经通过该事件增量更新图片列表
- 内置图片查看器已经支持：
  - 传入完整 URI 列表
  - 指定目标索引打开
  - 在 Activity 内维护 `uris` 和 `currentIndex`

当前缺失的能力：

- 没有“前台接收新图片自动显示”的配置项
- 没有暴露给前端的 Android 可见性状态
- 没有“查看器已打开时复用并跳转目标索引”的统一机制
- 没有 Android 专用自动打开协调层

---

## 3. 技术方案

### 3.1 核心思路

采用“**前端决策，Android 原生执行**”的方案：

- Android 原生继续在图片完成入库后发出已有的 `gallery-items-added`
- 前端新增 Android 自动打开协调逻辑，基于平台、配置、可见性和新增图片事件决定是否自动动作
- Android 原生新增轻量桥接能力：
  - 暴露 App 进程当前是否可见
  - 支持“打开或复用内置查看器并跳到目标图片”

这样可以最大化复用现有前端配置逻辑与图片列表状态，同时把 Activity 生命周期和查看器复用细节留在 Android 原生处理。

### 3.2 方案选择说明

对比过的方案：

1. **前端事件驱动（采用）**
   - 优点：复用现有配置与 `gallery-items-added` 事件链，改动集中，行为更易与设置 UI 保持一致
   - 缺点：需要额外补充查看器状态和可见性桥接

2. **Android 原生直接驱动查看器**
   - 缺点：查看器选择逻辑、设置逻辑、可见性逻辑会被拆散到前端与原生两侧，维护成本更高

3. **Rust 上传链路统一驱动**
   - 缺点：Android 图片真正可打开依赖 MediaStore / 扫描完成，Rust 侧不如现有 Android 事件链自然

因此采用方案 1。

## 4. 架构设计

```
┌─────────────────────────────────────────────────────────────────┐
│ Frontend (React + TypeScript)                                   │
│ ├─ ConfigCard                                                    │
│ │   └─ Android 图片查看设置：新增自动显示开关                       │
│ ├─ GalleryCard / gallery state                                  │
│ │   └─ 监听 gallery-items-added，维护完整图片列表                  │
│ └─ AndroidAutoOpenCoordinator（新增）                            │
│     ├─ 判断平台 / 配置 / 可见性 / 新图事件                         │
│     └─ 调用 ImageViewerAndroid.openOrNavigateTo(...)            │
└────────────────────────────┬────────────────────────────────────┘
                             │ JS Bridge
┌────────────────────────────┴────────────────────────────────────┐
│ Android Native (Kotlin)                                         │
│ ├─ MainActivity                                                 │
│ │   ├─ 维护 app visibility state                                │
│ │   └─ 注册 bridge / 向前端暴露 visibility 查询或更新事件            │
│ ├─ ImageViewerBridge                                            │
│ │   └─ openOrNavigateTo(uri, allUrisJson)                       │
│ └─ ImageViewerActivity                                          │
│     ├─ 未打开：正常启动                                          │
│     └─ 已打开：复用现有实例，更新数据源并跳到目标索引               │
└─────────────────────────────────────────────────────────────────┘
```

### 4.1 组件职责

#### A. 配置层

在现有 `androidImageViewer` 配置下新增布尔字段，用于控制：

- 当 App 可见且接收到新图片时，是否自动显示最新图片

该字段满足以下规则：

- 只在 `openMethod === built-in-viewer` 时显示
- 切换到 `external-app` 后在 UI 中隐藏
- 运行时只要 `openMethod !== built-in-viewer`，就算该值仍为 `true`，也必须按禁用处理

#### B. 前端自动打开协调层

新增 Android 专用协调逻辑，职责仅包括：

- 监听 `gallery-items-added`
- 从新增图片列表中取最后一张作为目标图片
- 读取当前完整图库列表，计算目标索引
- 检查自动动作前置条件
- 满足条件时调用原生桥接方法执行“打开或跳转”

该协调层不负责：

- 判断 Activity 是否已存在
- 维护原生 viewer 当前页面状态
- 直接管理 Android 生命周期

#### C. Android 原生桥接层

新增或扩展以下原生能力：

- App 可见性状态对前端可见
- 图片查看器支持单实例复用或等价语义
- 当查看器已存在时，更新 URI 列表并跳转到目标索引

## 5. 配置设计

### 5.1 配置模型

在现有 Android 图片查看配置中新增一个布尔字段，语义建议为：

- `auto_open_latest_when_visible`

其含义：

- 仅当 Android 使用内置查看器时生效
- App 进程可见时，接收到新图片后自动显示最新图片

### 5.2 默认值

建议默认值：`false`

理由：

- 这是新增行为，默认关闭更稳妥
- 避免已有用户在升级后收到意外弹出

### 5.3 配置界面行为

在现有 Android “图片查看设置”区域增加一个开关：

- 文案建议：`前台接收新图片时自动显示`

显示逻辑：

- `built-in-viewer`：显示
- `external-app`：隐藏

运行时生效逻辑：

- 只有 `openMethod === built-in-viewer && auto_open_latest_when_visible === true` 时，自动显示能力才可能启用

## 6. 数据流设计

### 6.1 新图片接收后的自动显示流程

```
文件上传完成
    │
    ▼
Android MediaStore 完成入库
    │
    ▼
emitWindowEvent("gallery-items-added", payload)
    │
    ▼
Gallery 前端先更新图片列表
    │
    ▼
AndroidAutoOpenCoordinator 收到新增图片事件
    ├─ 非 Android → 忽略
    ├─ 非内置查看器 → 忽略
    ├─ 自动显示开关关闭 → 忽略
    ├─ App 不可见 → 忽略
    └─ 满足条件 → 取最后一张新增图片为目标
    │
    ▼
调用原生 bridge：打开或跳转到目标图片
```

### 6.2 目标图片选择规则

当一次事件中包含多张新增图片时：

- 使用该批次中的最后一张作为目标图片

原因：

- 与用户要求一致
- 行为稳定、简单，不需要对一批图片逐张自动翻页

### 6.3 查看器打开 / 复用行为

统一采用“打开或跳转”的单语义原生接口：

- **查看器未打开**
  - 使用当前完整 URI 列表和目标索引打开内置查看器

- **查看器已打开**
  - 不再新开一个 Activity 层级
  - 将新的完整 URI 列表和目标索引发给现有查看器实例
  - 当前查看页直接跳到最新图片

### 6.4 不可见时的行为

如果新图片到达时 App 不可见：

- 不自动打开
- 不记录待处理队列
- 不在之后恢复可见时补开

这样能避免后台收到图片后打断用户当前上下文。

## 7. Android 可见性设计

### 7.1 判定标准

“可见”按 **进程可见** 处理，而不是“主界面是否完全占据前台”。

包括但不限于：

- 正常前台显示
- 分屏场景下仍可见
- 多窗口 / 浮窗场景下仍可见

### 7.2 暴露方式

Android 原生需为前端提供以下任一等价能力：

1. **可同步查询当前可见状态**
2. **可订阅可见性变化事件，并在前端维护最新状态**

本设计要求的只是语义，不强制具体 API 形态。实现时优先选择与当前 bridge 风格最一致的方案。

### 7.3 异常兜底

如果可见性状态暂时不可用或状态不可信：

- 默认按不可见处理
- 即“不自动打开”

---

## 8. ImageViewer 复用设计

### 8.1 目标

满足以下核心要求：

- 如果内置图片查看器已经在界面上，收到新图时应当直接翻页到新图
- 不能再叠加打开一个新的图片查看器页面

### 8.2 原生行为要求

`ImageViewerBridge` 需要支持一个“打开或导航到图片”的统一动作，语义如下：

- 若 `ImageViewerActivity` 当前不存在：启动它
- 若 `ImageViewerActivity` 当前已存在：
  - 更新其图片 URI 列表
  - 重新计算或接收目标索引
  - 直接切换到目标索引

### 8.3 更新策略

原生侧更新现有查看器时，应保证：

- 新增图片出现在最新列表中
- 当前索引最终落在目标图片上
- 对重复调用具备幂等性

即使短时间内连续收到多次新图事件，最终也应稳定落在最新收到的图片上。

---

## 9. 错误处理

### 9.1 前端侧

- 若新增图片事件没有合法可打开 URI：跳过自动显示
- 若目标图片在当前图库列表中无法定位：跳过自动显示
- 若当前模式为外部查看器：直接忽略自动动作

### 9.2 Android 原生侧

- 若复用现有查看器失败或实例已失效：回退为重新打开查看器
- 若收到不合法 URI 列表或非法索引：忽略此次导航请求并记录日志

### 9.3 防守式约束

即使前端误触发原生调用，原生也应再次校验调用参数，避免异常页面跳转。

---

## 10. 去重与触发约束

为避免重复自动打开，约束如下：

- 只使用 `gallery-items-added` 作为自动显示触发源
- 不再额外基于 `media-store-ready` 触发同等行为
- 一次 `gallery-items-added` 事件只触发一次自动动作
- 目标固定为该批次中的最后一张图片

---

## 11. 测试设计

### 11.1 前端测试

#### 配置 UI

- 使用内置查看器时显示自动显示开关
- 切换为外部查看器时隐藏自动显示开关
- 外部查看器模式下即使配置残留为 `true`，运行时仍不触发自动显示

#### 自动打开协调逻辑

- Android + 内置查看器 + 开关开启 + App 可见 → 触发自动显示
- 任一条件不满足 → 不触发自动显示
- 一次新增多张图片 → 选择最后一张

### 11.2 Android 原生测试

#### 可见性

- 可见生命周期内上报为 `true`
- 不可见生命周期内上报为 `false`
- 分屏 / 多窗口场景下保持“可见”为 `true`

#### 查看器复用

- 查看器未打开时：正常打开到目标索引
- 查看器已打开时：不新增页面层级，直接跳到目标索引
- 连续收到新图片时：最终停留在最新图片

### 11.3 集成验证

- Android + 内置查看器 + 开关开启 + App 可见：收到新图自动显示
- Android + 内置查看器 + 查看器已打开：收到新图直接翻页到新图
- Android + 外部查看器：配置项隐藏，功能不生效
- App 不可见时收到新图：不自动显示

## 12. 影响范围

预期会涉及以下区域：

- Rust 配置模型与默认配置
- TypeScript 配置类型导出 / 使用位置
- `ConfigCard` 配置界面
- Gallery 或相邻位置的 Android 自动打开协调逻辑
- Android `MainActivity` 可见性状态桥接
- Android `ImageViewerBridge` 与 `ImageViewerActivity` 的复用 / 跳转能力

本设计不包含以下扩展能力：

- 后台收图后的延迟补开
- 外部查看器模式下的自动打开
- 新图片自动播放 / 轮播
- 用户可配置“跳到最新”之外的其他策略

---

## 13. 验收标准

满足以下条件时，视为该功能完成：

1. Android 配置界面新增“前台接收新图片时自动显示”开关
2. 该开关只在内置查看器模式下可见
3. 内置查看器模式 + 开关开启 + App 可见时，收到新图后自动显示最新图片
4. 如果内置查看器已经打开，则直接翻页到最新图片，不新增一个新的查看器层级
5. 外部查看器模式下该配置项隐藏，且功能完全不生效
6. App 不可见时收到新图，不自动显示
